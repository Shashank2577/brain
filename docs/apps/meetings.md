# Meetings

## Purpose

Meetings is a Granola-style meeting lifecycle mini-app: prep notes before, live transcript during, AI summary + bullets + action items after. It is the user's working surface for the time around a calendar event — the calendar app owns the event itself; the meetings app owns everything else the user actually does with that block of time (typing rough notes while half-listening, capturing what was said, turning what was said into a clean summary and per-attendee follow-ups). The two-pane detail view (transcript left, AI summary + action items right) is the load-bearing UX — both sides update live, and the right pane is what the user reads after, shares to the team, and pastes into the linked note. Inter-app: calendar feeds upcoming events in, notes receives the summary, tasks receives one row per action item (per attendee), and the agent chat does all summarization. Meetings is the orchestrator that turns one calendar block into one note, one transcript, one summary, and N tasks.

## Data model

`templates/meetings/server/db/schema.ts`:

```ts
import {
  table,
  text,
  integer,
  now,
  ownableColumns,
  createSharesTable,
} from "@agent-native/core/db/schema";

export const meetings = table("meetings", {
  id: text("id").primaryKey(),
  title: text("title").notNull().default("Untitled meeting"),
  startsAt: text("starts_at"),
  endsAt: text("ends_at"),
  // Nullable FK to a calendar event. The calendar app owns the event row;
  // we only store the id (no DB-level FK — cross-template coupling).
  // Deleting the calendar event does NOT delete the meeting (see test plan).
  calendarEventId: text("calendar_event_id"),
  // High-level state for the right-pane spinner / CTA.
  status: text("status", {
    enum: ["scheduled", "live", "finalizing", "done", "failed"],
  })
    .notNull()
    .default("scheduled"),
  // User-typed pre-meeting / during-meeting notes (the left-pane Granola pad).
  // Plain text; markdown allowed but not required.
  prepNotes: text("prep_notes").notNull().default(""),
  // Nullable FK back to notes.create output (the published summary note).
  linkedNoteId: text("linked_note_id"),
  createdAt: text("created_at").notNull().default(now()),
  updatedAt: text("updated_at").notNull().default(now()),
  ...ownableColumns(),
});

export const meetingShares = createSharesTable("meeting_shares");

export const meetingTranscripts = table("meeting_transcripts", {
  id: text("id").primaryKey(),
  meetingId: text("meeting_id").notNull(),
  // Array of { startMs, endMs, speaker, text } as JSON. Stored as text for
  // dialect portability (Neon, SQLite, Turso).
  segmentsJson: text("segments_json").notNull().default("[]"),
  fullText: text("full_text").notNull().default(""),
  // Source priority is enforced when start-transcript chooses a path:
  // native > whisper > manual. See test plan.
  source: text("source", { enum: ["native", "whisper", "manual"] })
    .notNull()
    .default("manual"),
  status: text("status", {
    enum: ["pending", "streaming", "ready", "failed"],
  })
    .notNull()
    .default("pending"),
  createdAt: text("created_at").notNull().default(now()),
  updatedAt: text("updated_at").notNull().default(now()),
});

export const meetingAttendees = table("meeting_attendees", {
  id: text("id").primaryKey(),
  meetingId: text("meeting_id").notNull(),
  email: text("email"),
  name: text("name").notNull(),
  // organizer | required | optional — copied from the calendar event when synced.
  role: text("role", { enum: ["organizer", "required", "optional"] })
    .notNull()
    .default("required"),
  isOwner: integer("is_owner", { mode: "boolean" }).notNull().default(false),
});

export const meetingSummaries = table("meeting_summaries", {
  id: text("id").primaryKey(),
  meetingId: text("meeting_id").notNull(),
  // Three kinds of AI output, all generated by `meetings.finalize`. Stored
  // as separate rows so a user can regenerate one (e.g. action items) without
  // throwing away the others.
  kind: text("kind", { enum: ["summary", "bullets", "action_items"] }).notNull(),
  // Markdown for summary; JSON for bullets and action_items.
  content: text("content").notNull(),
  generatedAt: text("generated_at").notNull().default(now()),
  // Which model / route produced it — used by the eval pipeline.
  provider: text("provider"),
});

export const meetingFollowups = table("meeting_followups", {
  id: text("id").primaryKey(),
  meetingId: text("meeting_id").notNull(),
  // The natural-language action item ("Alice will write the spec by Friday").
  text: text("text").notNull(),
  // Email of the attendee responsible. Nullable when the AI couldn't assign.
  assigneeEmail: text("assignee_email"),
  // ISO date (YYYY-MM-DD) if the AI extracted a due date. Nullable.
  dueDate: text("due_date"),
  // Nullable FK to the tasks row created by `tasks.create`. NULL means the
  // followup hasn't been promoted to a task yet (or the promotion failed).
  taskId: text("task_id"),
  createdAt: text("created_at").notNull().default(now()),
});
```

All ownable tables (`meetings`) use `ownableColumns()` and route reads through `accessFilter(schema.meetings, schema.meetingShares)` (see `packages/core/src/sharing/access.ts`). Writes go through `assertAccess("meeting", id, "editor" | "admin")`. Transcripts, summaries, attendees, and followups are child rows — access is checked once on the parent meeting; child rows are loaded by `meetingId`. No `ownableColumns()` on children to avoid double-bookkeeping.

## Capabilities

All capabilities are `defineAction` exports under `templates/meetings/actions/`. Auto-mounted at `/_agent-native/actions/<name>` and registered as agent tools.

### `meetings.list`

List meetings the caller can see, split into upcoming and past relative to `now()`. Access role: viewer (via `accessFilter`).

```ts
input: z.object({
  filter: z.enum(["upcoming", "past", "all"]).default("all"),
  search: z.string().optional(),
  limit: z.number().int().min(1).max(200).default(50),
}).optional(),
output: z.array(z.object({
  id: z.string(),
  title: z.string(),
  startsAt: z.string().nullable(),
  endsAt: z.string().nullable(),
  status: z.enum(["scheduled", "live", "finalizing", "done", "failed"]),
  calendarEventId: z.string().nullable(),
  linkedNoteId: z.string().nullable(),
  attendeeCount: z.number(),
})),
```

### `meetings.get`

Single meeting + transcript + summary rows + attendees + followups. Access role: `viewer` via `resolveAccess("meeting", id)`.

```ts
input: z.object({ id: z.string() }),
output: z.object({
  meeting: z.object({ /* meetings row */ }),
  transcript: z.object({ /* meeting_transcripts row, nullable */ }).nullable(),
  summaries: z.array(z.object({ kind: z.enum(["summary","bullets","action_items"]), content: z.string(), generatedAt: z.string() })),
  attendees: z.array(z.object({ email: z.string().nullable(), name: z.string(), role: z.string(), isOwner: z.boolean() })),
  followups: z.array(z.object({ id: z.string(), text: z.string(), assigneeEmail: z.string().nullable(), dueDate: z.string().nullable(), taskId: z.string().nullable() })),
}),
```

### `meetings.create`

Create a meeting — ad hoc, or from a calendar event (in which case `calendarEventId` is set and attendees are copied from the event). No `assertAccess` (user creates own row); calls `calendar.get-event` when `calendarEventId` is provided so attendees + start/end are seeded.

```ts
input: z.object({
  title: z.string().min(1).max(200).optional(),
  startsAt: z.string().datetime().optional(),
  endsAt: z.string().datetime().optional(),
  calendarEventId: z.string().optional(),
  attendees: z.array(z.object({
    email: z.string().email().optional(),
    name: z.string(),
    role: z.enum(["organizer","required","optional"]).default("required"),
  })).optional(),
}),
output: z.object({ id: z.string(), status: z.literal("scheduled") }),
```

### `meetings.start-transcript`

Begin live transcription. Chooses a source by priority: `native` (desktop-app system-audio capture) → `whisper` (Whisper API on uploaded chunks) → `manual` (user types into a transcript pad). The chosen source comes from a `preferredSource` input or is auto-detected from the request context (desktop vs. web). Flips `meeting.status = "live"` and inserts a `meeting_transcripts` row with `status = "streaming"`. Idempotent — re-calling returns the existing transcript id. Access role: `editor`.

```ts
input: z.object({
  meetingId: z.string(),
  preferredSource: z.enum(["native","whisper","manual"]).optional(),
}),
output: z.object({ transcriptId: z.string(), source: z.enum(["native","whisper","manual"]) }),
```

### `meetings.stop-transcript`

Flips `meeting_transcripts.status` to `ready` (or `failed`) and `meeting.status` to `scheduled` (so the user can finalize). Access role: `editor`.

```ts
input: z.object({ meetingId: z.string() }),
output: z.object({ transcriptId: z.string(), status: z.enum(["ready","failed"]) }),
```

### `meetings.finalize`

The heavy capability. Reads the transcript + prep notes + attendees, delegates to the agent chat (via `sendToAgentChat` on the client or `writeAppState("notes-ai-request-...")` on the server — same delegation pattern as `meeting-notes/actions/enhance-notes.ts`) to generate three artifacts: a markdown summary, structured bullets, and a list of `{ text, assigneeEmail, dueDate }` action items. Then: inserts three `meeting_summaries` rows, inserts one `meeting_followups` row per action item, calls `notes.create` once with the summary as the body and stores the returned note id in `meeting.linkedNoteId`, and calls `tasks.create` once per followup (with `alsoNote: false` and `text: "<assigneeName>: <text>"`) storing the returned task id in `meeting_followups.taskId`. Flips `meeting.status = "done"`. Idempotent: rerunning replaces the previous summary + followup + task rows for the same meeting (best-effort cleanup of orphaned tasks via `tasks.delete` on the old ids). Access role: `editor`.

```ts
input: z.object({
  meetingId: z.string(),
  // Override the transcript text (rarely — for tests / replays).
  overrideTranscript: z.string().optional(),
}),
output: z.object({
  meetingId: z.string(),
  summaryNoteId: z.string(),
  taskIds: z.array(z.string()),
  followupIds: z.array(z.string()),
  provider: z.string().optional(),
}),
```

### `meetings.list-action-items`

Return the followup rows across all meetings the caller can see, with the linked task status joined in. Useful for "what do I owe people from last week's meetings" surfaces. Access role: viewer (filters via the parent meetings' `accessFilter`).

```ts
input: z.object({
  assigneeEmail: z.string().email().optional(),
  meetingId: z.string().optional(),
  includeCompleted: z.boolean().default(false),
}).optional(),
output: z.array(z.object({
  id: z.string(),
  meetingId: z.string(),
  meetingTitle: z.string(),
  text: z.string(),
  assigneeEmail: z.string().nullable(),
  dueDate: z.string().nullable(),
  taskId: z.string().nullable(),
  taskCompleted: z.boolean(),
})),
```

## UI surface

Routes: `/meetings` (list) and `/meetings/:id` (detail). Both file-routed under `app/routes/`.

- **`/meetings` list** — two collapsible sections: **Upcoming** (sorted by `startsAt` ascending) and **Past** (descending). Each row: title, time, attendee avatars (max 4 + overflow chip), status pill (`scheduled` / `live` / `done`), and a `link-2` icon when `linkedNoteId` is set that opens the linked note via `navigate({ app: "notes", noteId })`. Empty state: a single CTA "Connect your calendar" linking into onboarding when no calendar account is registered; otherwise "Create a meeting" as the primary CTA.
- **`/meetings/:id` detail** — two-pane layout. **Left pane:** transcript (scrolling segment list, each segment is `[speaker] text`, with a "Start notes" button at the top while `status = "scheduled"` that calls `meetings.start-transcript` and flips into a live streaming view). Below the transcript is the prep notes pad — a single contenteditable that writes to `meeting.prepNotes` via a debounced `meetings.update` (out of scope for this spec but listed for completeness). **Right pane:** AI summary (markdown rendered), bullets list, and an Action Items table (one row per followup with the linked task's checkbox state inline so completing here also completes the task via `tasks.complete`). When `status != "done"`, the right pane is empty with a single Finalize CTA that calls `meetings.finalize` and shows a spinner until `status = "done"`. A small "Linked event" pill at the top of the page links out to the calendar app: `navigate({ app: "calendar", eventId: calendarEventId })`.
- **Agent sidebar:** the agent reads `navigation` (`{ view: "meeting", meetingId, panel: "transcript" | "summary" }`) so the user can say "summarize this," "add an action item for Bob," or "rename this meeting" with the current meeting implied.
- All UI uses shadcn primitives (`Tabs`, `DropdownMenu`, `Sheet`, `AlertDialog`) and Tabler icons — no `window.confirm`, no inline SVG, no custom positioned popovers.

## Inter-app dependencies

- **`calendar.list-events`** — called by the list view to populate the "Upcoming" section when the user hasn't yet created a meeting for an event. Result is joined with the `meetings` table on `calendarEventId` so events already converted into meetings render with the meeting's status pill instead of "Not started."
- **`calendar.get-event`** — called by `meetings.create` when `calendarEventId` is provided. Result seeds `title`, `startsAt`, `endsAt`, and `attendees` on the new meeting row. If the call fails, the meeting is still created with whatever the user passed in directly (graceful degradation).
- **`notes.create`** — called once by `meetings.finalize` with `{ title: meeting.title + " — Notes", body: <markdown summary>, sourceRef: { type: "meeting", id: meetingId } }`. The returned `noteId` is written to `meeting.linkedNoteId`. The note is owned by the calling user (RPC runs as user, not as meetings app).
- **`tasks.create`** — called once per action item by `meetings.finalize` with `{ text: "<assignee>: <action>", alsoNote: false, dueDate: followup.dueDate, sourceRef: { type: "meeting", id: meetingId, followupId } }`. The returned `taskId` is written to `meeting_followups.taskId`. One task per followup, NOT one task per attendee unless the AI assigned multiple items to the same person — in which case each item is its own task (per-item, scoped to its assignee).
- **`tasks.complete`** — called by the right-pane Action Items table when the user toggles the inline checkbox; we proxy to tasks rather than maintaining a separate `meeting_followups.completedAt`.
- **Agent chat (delegation)** — `meetings.finalize` does NOT inline an LLM call. It writes a delegation request to `application_state["notes-ai-request-<meetingId>"]` (when called server-side) or calls `sendToAgentChat` (when triggered from the UI). The agent processes the prompt, calls back into `meetings.update-summary` (an internal sub-action) to write the three `meeting_summaries` rows, and then the original `finalize` call completes the task-fan-out step.

## Inter-app consumers

- **Calendar** — shows a "Has notes" badge on event rows when a meeting exists with matching `calendarEventId` and `meeting.status != "scheduled"`. Calendar calls `meetings.list({ filter: "all" })` and joins client-side; no shared DB access.
- **CRM** — contact timelines surface meetings as activities. CRM calls `meetings.list-action-items({ assigneeEmail: contact.email })` to render "what's outstanding with this person" on the contact page.
- **Notes** — the linked note carries a `sourceRef: { type: "meeting", id }` on creation, so opening the note shows a "From: <meeting title>" backlink that calls `meetings.get` to render the breadcrumb. Notes never reaches into the meetings DB directly.
- **Tasks** — tasks created from action items carry `sourceRef: { type: "meeting", id, followupId }`, so the Tasks inbox can group them under "From meetings."

All consumers use the public capability surface — no shared DB access, no direct imports across templates.

## Test plan

- [ ] **Unit — schema validation**: `meetings.create` rejects empty `title` (after defaulting), rejects malformed ISO datetimes; `meetings.finalize` rejects unknown `meetingId`.
- [ ] **Unit — `meetings.finalize` fan-out**: stubs the agent chat to return a known `{ summary, bullets, actionItems: [3 items] }`; asserts exactly 3 `tasks.create` calls were made (one per action item), each task's `ownerEmail` equals the meeting owner's email, asserts 1 `notes.create` call was made with the summary as the body, asserts the `meeting_summaries` row count equals 3 (one per kind), and `meeting.linkedNoteId` is the returned note id.
- [ ] **Unit — `meetings.finalize` per-attendee action items**: action items shaped `[{ assigneeEmail: "alice@…", text: "..." }, { assigneeEmail: "bob@…", text: "..." }, { assigneeEmail: "alice@…", text: "..." }]` produce 3 tasks (per-item, NOT 2 grouped per attendee), each text prefixed by the assignee name, `meeting_followups.taskId` set on each row.
- [ ] **Unit — `meetings.finalize` idempotency**: calling `finalize` twice in a row replaces the previous `meeting_summaries` rows (count still 3), best-effort calls `tasks.delete` for the previous task ids, and the second call's `taskIds` are all new.
- [ ] **Unit — `meetings.finalize` partial failure**: stubs `tasks.create` to throw on the 2nd call; asserts the meeting still flips to `status = "done"`, asserts the response carries a non-fatal warning listing the failed followup id, asserts `meeting_followups.taskId` is NULL for the failed one.
- [ ] **Unit — Transcription source priority**: with no `preferredSource` and a request context that exposes `desktopBridge: true`, `meetings.start-transcript` picks `native`; with `desktopBridge: false` and a Whisper API key configured, it picks `whisper`; with neither, it falls back to `manual`. Assert the inserted `meeting_transcripts.source` matches the priority order (`native > whisper > manual`).
- [ ] **Unit — Transcription source override**: `preferredSource: "manual"` always wins, even when the desktop bridge is present.
- [ ] **Integration — Linked-event cleanup**: create a calendar event, create a meeting referencing its `calendarEventId`, populate transcript + summaries, then `calendar.delete-event` the event. Assert the meeting row still exists, the transcript still exists, the summaries still exist, and `meeting.calendarEventId` is unchanged (orphaned-but-preserved — transcript is valuable). The calendar app does NOT cascade into meetings.
- [ ] **Integration — Linked-note cleanup**: delete the linked note via `notes.delete`. Assert the meeting row still exists and `meeting.linkedNoteId` becomes `NULL` (handled by a sweeper job or on next `meetings.get`). The summary content in `meeting_summaries` is preserved.
- [ ] **Integration — Access scoping**: user B cannot read, list, finalize, or list-action-items for user A's meetings; assertions hit `ForbiddenError` for each verb; `meetings.list` returns 0 for user B.
- [ ] **Integration — `meetings.list-action-items` join**: seeds 2 meetings, 5 followups across them, marks 2 of the linked tasks complete; with `includeCompleted: false` returns 3, with `includeCompleted: true` returns 5, with `assigneeEmail: "alice@…"` returns only Alice's rows.
- [ ] **E2E — Full lifecycle**: Playwright creates a meeting from a calendar event (asserts attendees are copied), clicks Start notes (asserts transcript pane goes live), types into the prep pad (asserts `prepNotes` saved), clicks Stop, clicks Finalize, waits for `status = "done"`, asserts the right pane renders the markdown summary, asserts 3 task rows exist in `/tasks` with the meeting title as a source pill, asserts the linked note exists at `/notes/<id>` with the same summary.
- [ ] **E2E — Agent flow**: agent runs `meetings.create({ calendarEventId: "evt_…" })`, `meetings.start-transcript`, `meetings.stop-transcript`, `meetings.finalize`; asserts the UI (polling) updates within 2s at each step without manual refresh; agent then runs `meetings.list-action-items({ assigneeEmail: caller })` and asserts the returned items match the followups.

## Migration from manifest

Delete `packages/fluid-os/examples/apps/meetings/manifest.ts`. The in-memory `Map<string, Recording | Transcript | MeetingNote>` is discarded — no data migration needed; the manifest was demo state only.

**Source recommendation: start from `templates/meeting-notes/` (not fresh).** Strong reasons:

1. `meeting-notes/server/db/schema.ts` already defines the core tables we need (`meetings`, `meeting_transcripts`, `meeting_notes`, `meeting_attendees`, `meeting_shares`) with `ownableColumns()` already wired and the same `accessFilter` pattern. Starting fresh would re-derive all of this.
2. `meeting-notes/actions/enhance-notes.ts` already implements the delegate-to-agent pattern (`writeAppState("notes-ai-request-…")`) that `meetings.finalize` needs. Reusing it preserves a known-good integration with the agent chat.
3. `meeting-notes/actions/create-meeting.ts` already does the upsert-attendees + extract-companies-from-email-domains work; we'd otherwise rebuild it.
4. The template ships with people/companies/folders that the manifest doesn't model — these are valuable for the CRM-style consumer surface and we shouldn't throw them away.

What changes vs. `meeting-notes`:

- Rename the template directory `templates/meeting-notes/` → `templates/meetings/` (atomic — update `packages/shared-app-config/templates.ts` slug and `netlify.toml`).
- Flip `hidden: true` → `hidden: false` in `packages/shared-app-config/templates.ts` AND `packages/core/src/cli/templates-meta.ts`. Add `meetings` to the homepage, docs sidebar, and CLI catalog per the allow-list rules in `AGENTS.md`.
- Add the new `meeting_summaries` and `meeting_followups` tables (the `meeting-notes` template only stores enhanced notes as a single `meeting_notes.enhancedContent` text blob; we want three structured rows for summary / bullets / action items, plus a per-followup row that can carry a `taskId` FK).
- Add `meetings.finalize` as a new action wrapping the existing `enhance-notes` flow, plus the `tasks.create` and `notes.create` fan-out from the old manifest's `process-recording`.
- Add `meetings.list-action-items` as a new action — no equivalent exists today.
- Drop the `recipes` table from `meeting-notes` (multi-meeting AI prompts are a feature for a later release; not needed for v1 of the meetings app).
- Preserve from the manifest's public surface: capability names (`meetings.list`, `meetings.create`, `meetings.finalize`) and the `consumes: ["calendar.list-events", "content.create-document", "tasks.create", "mail.find-contact"]` contract — existing demo callers reference these. Update `packages/fluid-os/examples/apps/index.ts` to re-export from the new template's capability surface instead of the manifest.

A schema migration runs additively (per the no-breaking-changes rule in `AGENTS.md`): the new `meeting_summaries` and `meeting_followups` tables are added; existing `meeting_notes.enhanced_content` is preserved unchanged and is backfilled into a `summary` row on first read of any meeting whose `enhanced_content` is non-null and whose `meeting_summaries` rows are empty.
